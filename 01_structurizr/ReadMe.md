# ДЗ 01 – Документирование архитектуры в Structurizr (Вариант 21)

## Вариант 21 – Система управления арендой автомобилей (Hertz-подобный сервис)

### Описание системы

Система управления арендой автомобилей позволяет клиентам арендовать автомобили, а менеджерам — управлять автопарком. Система обеспечивает полный цикл аренды: от поиска доступного автомобиля до завершения аренды и оплаты.

Пользователи взаимодействуют с системой через веб-приложение. Запросы поступают через API Gateway и далее обрабатываются специализированными сервисами.

**Основные подсистемы:**
- сервис управления пользователями
- сервис управления автопарком
- сервис бронирования
- платежный сервис
- сервис уведомлений

Для хранения данных используются реляционные базы данных (по одной на каждый сервис).

Система интегрируется с внешними сервисами:
- платежный шлюз для обработки транзакций
- SMS-сервис для отправки уведомлений

### Основные API системы

Реализуются следующие операции:

- создание нового пользователя
- поиск пользователя по логину
- поиск пользователя по маске имени и фамилии
- добавление автомобиля в парк
- получение списка доступных автомобилей
- поиск автомобилей по классу
- создание аренды
- получение активных аренд пользователя
- завершение аренды
- получение истории аренд

### Роли пользователей

В системе предусмотрены следующие роли:

**Гость (Guest)**
Незарегистрированный пользователь. Может просматривать список доступных автомобилей и регистрироваться в системе.

**Клиент (Customer)**
Зарегистрированный пользователь. Может:
- создавать новые аренды автомобилей
- просматривать свои активные аренды
- завершать аренды
- просматривать историю своих аренд

**Менеджер автопарка (Fleet Manager)**
Сотрудник компании, управляющий автопарком. Может добавлять новые автомобили в парк.

**Администратор (Admin)**
Пользователь с полными правами. Может выполнять все операции:
- создание и поиск пользователей
- просмотр доступных автомобилей и поиск по классу
- создание, завершение и просмотр аренд (любых пользователей)
- просмотр истории аренд

### Внешние системы

**Платежный шлюз (Payment Gateway)**
Внешняя система для обработки платежей. Используется для блокировки и списания средств при аренде.

**SMS-сервис (SMS Service)**
Внешний сервис для отправки SMS-уведомлений (подтверждение аренды, напоминания о возврате, чек об оплате).

### Архитектура системы

Архитектура системы описана в стиле C4 Model с использованием Structurizr DSL.

Описание архитектуры находится в файле `workspace.dsl`.

### Диаграммы

#### C1 — System Context

Диаграмма контекста системы показывает:
- роли пользователей (Guest, Customer, Fleet Manager, Admin)
- взаимодействие пользователей с системой через веб-приложение
- интеграцию системы с внешними сервисами (Payment Gateway, SMS Service)

#### C2 — Container Diagram

Диаграмма контейнеров показывает основные компоненты системы:

- **Web Application** — одностраничное приложение для всех ролей
- **API Gateway** — единая точка входа, маршрутизация запросов, аутентификация
- **User Service** — управление пользователями (регистрация, поиск)
- **Fleet Service** — управление автопарком (добавление автомобилей, поиск)
- **Booking Service** — управление арендами (создание, завершение, история)
- **Payment Service** — интеграция с платежным шлюзом
- **Notification Service** — отправка SMS-уведомлений
- **Базы данных** — отдельная БД для каждого сервиса (User DB, Fleet DB, Booking DB, Payment DB)

На диаграмме отображены технологии контейнеров и протоколы взаимодействия.

#### Dynamic Diagrams

**D1 — Создание аренды (Create Rental)**

Динамическая диаграмма показывает архитектурно значимый сценарий создания новой аренды:

1. Клиент выбирает автомобиль и даты в веб-приложении
2. Запрос проходит через API Gateway в Booking Service
3. Booking Service валидирует пользователя через User Service
4. Проверяется доступность и резервируется автомобиль через Fleet Service
5. Запрашивается авторизация платежа через Payment Service (с вызовом внешнего Payment Gateway)
6. Сохраняется запись об аренде в базе данных
7. Отправляется команда на отправку SMS-уведомления
8. Клиент получает подтверждение

**D2 — Завершение аренды (Complete Rental)**

Динамическая диаграмма показывает сценарий завершения активной аренды:

1. Клиент запрашивает завершение аренды
2. Booking Service обновляет статус аренды в базе данных
3. Запрашивается финальное списание средств через Payment Service
4. Отправляется SMS-чек с деталями аренды
5. Клиент получает подтверждение

### Используемые технологии

| Компонент | Технология |
|-----------|------------|
| Web Application | Vue.js |
| API Gateway | Nginx |
| Backend микросервисы | FastAPI |
| Базы данных | PostgreSQL |
| Message Queue | RabbitMQ |
| Планировщик задач | Встроенный в Booking Service (Quartz) |
| Аутентификация | JWT |
| Взаимодействие сервисов | HTTPS/REST, Message Queue |
| Взаимодействие с внешними системами | HTTPS/REST (API Key) |
